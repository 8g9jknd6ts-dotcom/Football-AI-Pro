# Football AI Pro｜模型与规则摘要

**版本：Football AI 1.0.0-alpha.48**
**更新：2026-07-19**  
**用途：** 本文是当前分析、回测、报告与数据维护的统一执行摘要；与完整规则库、数据字典和回测记录配套使用。

## 一、当前结论与边界

1. 系统当前处于**研究与验证阶段**，没有任何模型被标记为 production（正式盈利模型）。
2. 所有赛前结论均为数据研究输出，不保证单场命中，更不能用「高 AI 评分」代替胜率或收益承诺。
3. 市场赔率是重要基准：热门方向通常有较高命中率，但只有在可复现回测中同时验证**赔率价值、ROI、样本量和稳定性**后，才能成为正式推荐。
4. 赛后比分、赛况和统计只能用于复盘、结算、模型评估；**不得回灌到该场赛前特征**，防止未来信息泄漏。

## 二、数据分层与可信度

| 层级 | 数据类型 | 可用范围 | 限制 |
|---|---|---|---|
| A | 可复核的官方或结构化历史赛果 | 标准化数据库、审计回测 | 必须保留来源和 MatchID |
| B | 用户提供并确认的球探/竞彩截图 | 当前比赛的盘口与报告证据 | 只代表截图时点，不自动当作连续走势 |
| C | 第三方候选来源、无法交叉验证的抓取 | 研究线索 | 不进入正式回测或正式推荐 |

**多公司赔率规则：** 只有在保留公司名称、盘口、初盘/即时盘、时间点与方向的情况下，才可判断「多家公司同步」。只有平均值或单家公司变化时，不得伪装为同步资金信号。

## 三、统一数据结构

- 每场历史比赛必须有唯一 `MatchID`，并标准化联赛、赛季、开赛时间、主客队、赛果与数据来源。
- 1X2（胜平负）、竞彩让球胜平负、亚洲让球盘、大小球盘必须分开记录与评估，不允许混用盘口含义。
- 中国竞彩的「让球胜平负」与亚洲盘不是同一个市场：报告必须分别写出让球数和推荐方向。
- 预测、赔率快照、推荐、实际结果、结算和盈亏，以及用户修正意见，均应写入统一跟踪数据库；不得只依赖对话记忆。

## 四、模型状态

### 竞彩让球胜平负独立层

- 标准胜平负模块不得直接推出让球结论；让球由 `HandicapModel.psm1` 独立建模。
- 让球层先输出净胜球分布，再按实际让球数映射让胜/让平/让负。
- 净胜球分布只可由开赛前、带可解析比赛日期的历史赛果计算；无日期历史不得作为回退样本。输出至少包含主胜1球、主胜2球、主胜3球以上、平局、客胜五档，且五档概率之和必须为 100%。
- 报告必须明确标记净胜球分布为“可用”或“不可用”。不可用时，模型概率、edge、置信度均留空，并以 `HANDICAP_DECISION=SKIP` 结束；不得用赔率方向补写概率。
- 输出必须含模型概率、市场隐含概率、edge、置信度、触发或拒绝原因。
- 数据不足、盘口冲突、优势不足或历史门禁未通过时，必须 `HANDICAP_DECISION=SKIP`。
- 让球回测按联赛、实际让球线与时间顺序切分；近期赛果仅审计，不调参。

| 模块 | 当前用途 | 正式状态 |
|---|---|---|
| 欧洲 1X2 市场基准 | 概率/命中率比较基准 | 研究基准，不是自动推荐 |
| 泊松 / Dixon-Coles | 比分、总进球候选分布 | 候选，需联赛级回测 |
| 竞彩返还率修正 | 比较竞彩与市场赔率 | 候选，需可复算赔率快照 |
| 凯利指数 | 风险与赔率价值辅助 | 仅辅助，不单独定方向 |
| 亚洲让球与大小球 | 盘口方向研究 | 候选，必须独立回测 |
| 冷门猎手 1.0 | 识别市场与基本面分歧 | 研究模块，未准入正式模型 |
| 庄家行为 / 多公司同步 | 盘口轨迹与一致性研究 | 数据尚未完整，禁止过度解释 |

## 五、关键规则（现行摘要）

- **Rule001–Rule006：** 建立 MatchID、统一字段、原始数据只读、标准化入库和来源可追溯。
- **Rule009–Rule010：** 报告必须说明数据缺口与模型状态，禁止用估算数据伪装实盘事实。
- **Rule012：** 赛后必须记录实际结果与结算。
- **Rule013–Rule016：** 命中率、市场基准、赔率价值与 ROI 必须分开评估；正式盈利结论以 ROI 和稳定性为门槛。
- **Rule019：** 低质量、重复或异常样本不得进入正式回测。
- **Rule021–Rule024：** 赛前与赛后数据隔离；截图盘口需明确初盘/即时盘与球探符号映射。
- **Rule028：** 未经交叉验证的第三方历史赔率只能作为候选上下文。
- **Rule029–Rule031：** 初盘和即时盘分列；数据不足或信号冲突时必须允许弃赛。
- **Rule057–Rule060：** 中文截图的队名、盘口、赔率与市场类型需显式解析，禁止凭符号猜测。
- **Rule061–Rule066：** 联赛样本质量、重复键、单公司覆盖、结构有效性与来源真实性分别审核；让球盘单独评估。
- **Rule067–Rule071：** 复盘须保留预测快照、实际赛果、结算、盈亏、偏差原因与模型修正记录。

## 六、赛前分析流程

1. 确认比赛身份、联赛、开赛时间和主客队，生成或匹配 MatchID。
2. 录入竞彩胜平负与让球胜平负；分别注明让球数。
3. 录入用户截图中的多家公司欧赔、亚盘和大小球初盘/即时盘，并保留证据时间。
4. 结合可用的联赛历史样本、对战样本、主客表现和赛程信息；缺失内容明确写为「未验证」。
5. 先比较市场隐含概率，再用比分/总球候选模型做交叉验证。
6. 输出方向、比分、总进球、冷门风险、推荐赔率、AI 评分和信心指数；若门禁不通过，明确标记「弃」。
7. 赛后只做复盘与数据库结算，不修改已经记录的赛前预测。

## 七、报告字段解释

- **AI 评分：** 当前数据、模型与盘口信号的一致性评分，不等同于单场真实胜率。
- **信心指数：** 证据完整度、方向一致性和风险控制后的相对把握度，不是保本承诺。
- **冷门指数：** 市场热门方向被反向风险、样本不足或盘口异常削弱的程度。
- **推荐赔率：** 仅用于评估赔率价值；赔率低并不自动等于值得串关。
- **弃：** 数据缺口、市场分歧、模型未通过门禁或赔率与概率不匹配时的正式结论，不是漏分析。

## 八、正式准入门禁

一个模块要进入正式模型，至少需要：

1. 来源与字段通过数据审计；
2. 足够的、按时间切分的历史样本；
3. 与市场基准的公平比较；
4. 命中率、校准度、赔率价值、ROI、最大回撤和稳定性均有记录；
5. 没有未来信息泄漏；
6. 完成版本、规则、CHANGELOG、测试与回测记录。

在门禁全部通过前，系统默认保持 `FormalRecommendation=false`。

## 九、维护约定

- 新增规则必须按 `RuleXXX` 编号，进入规则库并记录版本日志。
- 每次分析后更新用户偏好、预测记录、赔率快照、实际结果和盈亏。
- 每次代码或文档更新后，检查改动、更新 README / CHANGELOG / VERSION、运行相关测试或回测、扫描密钥与大文件，再提交并推送。
